# Reflection: Assignment 5

## Challenges in Translating Requirements into Use Cases and Test Cases

One of the main challenges encountered during this assignment was translating the functional and non-functional requirements from the System Requirements Document (SRD) into clear and structured use cases. While the requirements defined what the system should do, they did not always explicitly describe how users would interact with the system. This required careful interpretation to ensure that each use case accurately represented real user behavior while still aligning with the original requirements.

Another challenge was maintaining consistency and traceability between Assignment 4 and Assignment 5. Each use case and test case needed to map directly to a specific functional requirement (e.g., FR1–FR10). Initially, it was difficult to ensure that no requirement was overlooked or duplicated. For example, some requirements such as booking and notifications (FR2 and FR7) are closely related and had to be combined logically within a single use case without losing clarity. Ensuring that all requirements were covered while avoiding redundancy required careful planning.

Designing alternative flows for use cases also proved to be challenging. While basic flows are straightforward, identifying realistic error scenarios (such as unavailable time slots or failed payments) required thinking beyond the “happy path.” This step was important to ensure that the system design is robust and accounts for real-world situations, but it added complexity to the documentation.

Developing test cases introduced another layer of difficulty. Each test case had to validate a specific requirement and include clear steps and expected results. Ensuring that the expected results aligned with the acceptance criteria in the SRD (such as response times and success rates) required attention to detail. Additionally, designing non-functional test cases was more complex than functional ones, as they involve system qualities like performance, security, and usability rather than direct user actions.

Another key challenge was balancing detail and clarity. Providing too much detail in use case specifications could make them difficult to read, while too little detail could make them incomplete. Achieving the right level of abstraction required multiple revisions.

Finally, aligning the work with stakeholder concerns highlighted in the stakeholder analysis required careful consideration. For example, ensuring fast booking processes and system performance directly supports student and IT stakeholder needs. This reinforced the importance of designing use cases and test cases that not only meet technical requirements but also address real user expectations.

Overall, this assignment improved my understanding of how to bridge the gap between requirements and system design. It demonstrated the importance of traceability, clarity, and thorough testing in developing a reliable and user-centered system.
